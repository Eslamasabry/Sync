# Flutter Reader Build-vs-Buy Decision

## Decision

For MVP, build a custom Flutter reader on top of the backend-owned reader model.

Do not use a full off-the-shelf EPUB reader package as the primary rendering engine.

## Why

This project does not need a general EPUB viewer first. It needs deterministic token rendering tied to backend alignment output.

Packaged EPUB readers are optimized for:

- opening EPUB files directly in the app
- pagination and chapter navigation
- highlights and annotations at reader level
- general reading UX

This project needs:

- a canonical reader model generated by the backend
- stable token indices
- backend-to-client location mapping
- cheap word-level highlight updates during playback
- no runtime re-tokenization drift

Those needs are closer to a custom text renderer than a generic EPUB viewer.

## Evaluated Options

### `flureadium`

Use as reference only.

Pros:

- most complete Flutter reading stack found
- built on Readium
- supports EPUB 2/3, audiobook playback, highlights, reading preferences, and a `ReaderWidget`

Cons:

- heavy reading framework
- pushes the app toward Readium's rendering and locator model
- harder to make backend token timing the source of truth
- LGPL-3.0 may be fine, but it is still a larger licensing and integration choice than MVP needs

Verdict:

- not for MVP core renderer
- useful reference for navigation, locator handling, and reading preferences

### `epub_view`

Use only for spikes or code reference.

Pros:

- pure Flutter widget approach
- exposes EPUB CFI support
- easier to inspect and fork than WebView-based readers

Cons:

- still a general EPUB rendering package, not a token-accurate synced reader
- package activity appears older than other options
- built around rendering EPUB content directly, which conflicts with the backend-owned reader model

Verdict:

- acceptable for short experiments
- not the primary architecture

### `flutter_epub_viewer`

Reject for MVP core renderer.

Pros:

- fast to demo
- chapter navigation, selection, and highlighting already exist

Cons:

- built on Epub.js and `flutter_inappwebview`
- requires WebView setup and Android cleartext-related configuration
- WebView/Epub.js stack makes deterministic backend-driven token highlighting harder
- pub metadata and README license text should be double-checked before adoption

Verdict:

- not aligned with project constraints

### `Iridium`

Use as architecture reference only.

Pros:

- large open-source Flutter e-reader SDK
- supports EPUB, audiobook, highlights, TTS, media overlays, themes, and navigation
- proven reader concepts and app structure

Cons:

- large framework surface area
- would pull the project toward a full reader SDK instead of the custom synced-reader architecture
- more than MVP needs

Verdict:

- useful reference repo
- not the core dependency

### OpenLeaf / `FlutterEbookApp`

Use as app-structure reference only.

Pros:

- real Flutter reader app
- Riverpod-based app structure
- demonstrates a practical open-source reading app shell

Cons:

- uses Iridium as the reader layer
- not a sync-first architecture

Verdict:

- good reference for app organization and product flow
- not a direct dependency

## Final Recommendation

### Build

Build these pieces in-house:

- reader model DTOs
- token-aware reader rendering
- active token lookup by playback time
- phrase-level smooth scroll behavior
- tap-to-seek mapping from token to audio time

### Reuse

Reuse targeted libraries for narrow jobs:

- `just_audio` for playback
- `audio_session` for media session behavior
- `flutter_riverpod` for state
- `dio` for API and downloads
- `web_socket_channel` for progress updates
- `scrollable_positioned_list` or equivalent for indexed reading navigation
- `google_fonts` for theme typography

### Borrow Ideas From

- `flureadium`
- `Iridium`
- `OpenLeaf`
- `epub_view`

Borrow patterns, not the renderer.

## Effect on Implementation

This decision confirms:

- the backend owns EPUB parsing
- the backend exports the reader model
- Flutter does not parse EPUB directly in MVP
- `sync.json` points into the reader model, not raw HTML content

## Revisit Conditions

Revisit this decision only if one of these becomes true:

1. The custom reader renderer becomes the main schedule risk.
2. A Flutter reader package appears that supports stable token-level external control without WebView indirection.
3. The product expands from sync-first reading into a general-purpose EPUB reader platform.
